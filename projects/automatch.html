Overview
What it
solves.

Buying a car in Malaysia is complicated. Hundreds of models, varying loan rates, family constraints, brand preferences — most people either overspend or settle. Automatch was built to fix that: input your financial profile and preferences, get ranked recommendations with reasoning.

The system separates concerns cleanly — PHP handles the web layer, user auth, profiles, and rendering, while a dedicated Flask API handles all recommendation logic. The API accepts a JSON payload, runs a weighted scoring algorithm across the vehicle database, and returns ranked results with per-car reasoning.

Looking back, this is exactly what an agentic pipeline does: a user intent triggers a reasoning engine that profiles, scores, filters, and explains — autonomously. I built this before I had the vocabulary for it.

Scoring
How cars
get ranked.

Each car is scored across four dimensions, then combined into a weighted total. The weights reflect what actually matters when buying a car in Malaysia — financial fit first, practicality second, then brand loyalty and features.

Financial fit uses amortization math — monthly payment calculated from price, interest rate, loan tenure, and down payment percentage. A car is financially scored 0–1 based on how it falls within the user's disposable income cap.

Challenges
What was
hard.

Three engineering problems that weren't obvious at the start — and taught the most.

01
The Affordability Cap Problem
What counts as "affordable"? The first version used a hard monthly cap — any car above the limit was rejected outright. But users with conservative income inputs would get zero results and no explanation. The edge case was common: someone earning RM4,000/month after expenses had a calculated cap so low that even the cheapest cars were excluded.
Added a 25% tolerance buffer on auto-calculated caps, and inverted the amortization formula to derive a maximum sticker price from a monthly payment target — so the filter always works with the user's real budget ceiling, not an arbitrary cutoff.
02
Spec Data is a Mess
The vehicle_specifications table stores car data scraped from multiple sources — which means the same field appears under different keys. Seating capacity might be stored as "seats", "Seats", "seating capacity", or "Seat" depending on the source. Same for horsepower ("hp", "Horsepower", "power") and top speed. A naive lookup returns NULL for half the fleet.
Wrote nested COALESCE queries with category-aware fallback chains — prefer the canonical category/key pair first, then any matching key name, then a direct exact match. No source data cleanup required.
03
Brand Preference Graceful Degradation
If a user preferred Proton + SUV but no affordable Proton SUVs existed in the database, the original system returned an empty result set with no explanation. Users assumed the system was broken. The problem was that strict filtering and user preference were treated as equally hard constraints.
Rewrote filtering as a four-level degradation: try brand + config → brand only → config only → open pool. Each fallback level is tracked so the response always explains why the results differ from the stated preference.

What's
next.

The current engine is rule-based — intentional for V1. The roadmap moves toward a model that learns from user behaviour and explains its decisions.

Rule-based recommendation engine
Weighted scoring across financial fit, practical needs, brand preference, and features. Amortization-based monthly payment calculation.

Multi-profile system with demographic inputs

Session & impression tracking
Every recommendation session is logged — which cars were shown, in what position, with what feature snapshot. Foundation for future ML training data.

ML-based recommendation engine
Replace rule-based scoring with a trained model using the accumulated session/impression data. Target: XGBoost or LightGBM for interpretability.
